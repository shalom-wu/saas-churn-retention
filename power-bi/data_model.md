# Data Model

All tables load from `data/powerbi/` (regenerated by
`python scripts/run_sql.py`) via the `DataFolder` parameter.

```
┌──────────────────────────────┐ *      1 ┌──────────────────────────────┐
│   fact_customers (7,043)     ├──────────┤   ltv_by_contract (3)        │
│  customerID · demographics   │ Contract │  hazard · expected lifetime  │
│  tenure · tenure_band        │          │  avg_ltv · cost_per_churn    │
│  Contract · PaymentMethod    │          │  (computed by src/ltv.py —   │
│  InternetService · add-ons   │          │   discounting stays in       │
│  MonthlyCharges · churn_flag │          │   Python on purpose)         │
└──────────────────────────────┘          └──────────────────────────────┘

Disconnected reference tables (no relationships needed):
  at_risk_segments (4)     — the four named segments, sized in MRR
  kpi_cost_of_churn (1)    — headline cost numbers from the LTV model
  kpi_churn_by_addons (7)  — add-on gradient, internet customers only
  dim_interventions (3)    — the costed options A/B/C (assumption table)
```

## The Python ↔ SQL handoff (deliberate)

SQL computes everything that is counting and rates —
`fact_customers` and the KPI aggregates come from `sql/kpi_views.sql`.
Python computes everything that needs finance math — discounted LTV,
cost-per-churn, and the cohort cost summary come from `src/ltv.py`. The
exporter (`scripts/run_sql.py`) writes both into one folder so Power BI has
a single documented input set.

`tenure_band` is zero-padded in the export (`00-06m`, `07-12m`) purely so
Power BI's alphabetical category sort shows life-cycle order.

## Notes

- `dim_interventions` is an **assumption table** — the deck's costed
  options, not data. Labeled as such on the Value & Action page.
- No date table: the source is a snapshot with no event dates, so time
  intelligence doesn't apply.
